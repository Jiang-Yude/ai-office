# 環境與分工路由

這份文件回答兩件事：
- 目前在哪一層工作
- 這次誰該做主力、誰該做外部視角、誰該收斂

## 一、先判斷你在哪一層

### 知識層

適用情況：
- 只是要決定怎麼拆工
- 只是要教學、說明、規劃
- 當前環境不能跑 shell

這一層要產出的是：
- 任務類型判斷
- 分工建議
- 手動操作順序

### 工具層

適用情況：
- 需要真的跑多個 provider
- 需要保留原始輸出與 logs
- 需要生成材料包給後整合器

這一層才進 `scripts/cross-check.sh`。

## 二、三個角色

### 1. 主力

主責：
- 理解任務脈絡
- 做核心判斷
- 決定這次到底要不要調度其他模型

適合的工具：
- Claude Cowork
- Claude Code

### 2. 外部視角

主責：
- 刻意不帶太多內部脈絡
- 只看局部文件
- 用距離感抓自說自話、結構問題、可讀性問題

適合的工具：
- Codex
- GPT 類模型

### 3. 收斂者

主責：
- 讀完原始材料
- 區分共識與獨見
- 做最後輸出

適合的工具：
- Cowork：保留知識庫脈絡
- Code：快速整理與一次性分析

## 三、常見路由判斷

### 需要知識庫脈絡

主力優先放在 Cowork 或能讀完整專案脈絡的工具。

### 需要公開資料搜尋

把搜尋拆出去，不要消耗主力模型的額度。

常見選項：
- Codex CLI 子代理搜尋
- 其他可控的外部搜尋工具

### 需要第二雙眼睛

不要再叫另一個懂全部脈絡的模型重做一遍。
反而要刻意給它較少背景，讓它從外部視角看。

### 只要一次性比較

不一定要 Cowork。很多情況用 Code 或 Codex 做快速收斂就夠。

## 四、實用對應表

| 角色 | 常見工具 | 用法重點 |
|------|---------|---------|
| 主力 | Claude Cowork | 知識庫脈絡、最終判斷、正式落檔 |
| 主力 | Claude Code | 終端機調度、判斷、實作 |
| 外部視角 | Codex | 單檔檢查、距離感、交叉比對 |
| 搜尋研究 | Codex CLI 子代理 | 公開資料搜尋、方案整理 |
| 收斂者 | Cowork | 要脈絡、要延續 Obsidian |
| 收斂者 | Code | 要快速整理、偏技術任務 |

## 五、最容易犯的錯

### 錯一：把所有模型都叫來重做

這通常不是協作，是重複勞動。

### 錯二：把搜尋、判斷、實作綁死在同一模型

結果通常是：
- 額度浪費
- 上下文混亂
- 最後不好切換工具

### 錯三：忘記誰該看全貌、誰該保留距離

真正有效的多 AI 協作，常常是故意讓其中一個模型不知道太多。
