# 模式與後整合器

## 三種模式

### `outside`

用途：
- 教案
- 文章
- 企劃
- 任何想知道外人會不會看不懂的文件

邏輯：
- 主力先寫
- 第二個模型刻意不帶背景
- 只根據文件本身找卡點

重點不是重寫全文，而是指出外部讀者會卡住哪裡。

### `style`

用途：
- 標題發想
- 文案改寫
- 語氣比較
- 同素材多版本輸出

邏輯：
- 同一份素材
- 同一個任務
- 不同模型各出一版

這個模式不要硬選唯一正解。
它的價值是把風格差異攤開來。

### `blindspot`

用途：
- 重要會議記錄
- 策略文件
- 高風險決策備忘
- 想知道有哪些自己沒看到的假設

邏輯：
- 多個模型各自找盲點
- 最後整理成共識與獨見

這個模式最重要的是：
- 兩家都提到的，優先處理
- 只有一家提到的，不一定錯，但值得想

## 什麼時候選哪個模式

如果你要的是：
- 外部讀者視角 → `outside`
- 多種表達版本 → `style`
- 風險與遺漏盤點 → `blindspot`

## 兩種分析強度

### `light`

用途：
- 一般企劃初審
- 草稿抓漏
- 想先快速看三家有沒有共識

邏輯：
- 每家模型先各做一輪
- 最後交給整合器做總結

### `heavy`

用途：
- 重要決策
- 高風險企劃
- 比賽方向取捨
- 想保留「模型如何看彼此」的推理痕跡

邏輯：
- 每家模型先各做一輪
- 再讓每家模型各自分析三家第一輪分析的異同
- 最後交給整合器做總結

選擇原則：
- 快速判斷先用 `light`
- 要把分歧攤開並保留第二層思辨時用 `heavy`

## 三種後整合器

### `cowork`

適合：
- 要保留知識庫脈絡
- 要延續既有 Obsidian 工作流
- 要做正式落檔
- 要把結果接回個人系統

### `code`

適合：
- 快速整理
- 一次性比較
- 純文字比對
- 偏技術任務

### `manual`

適合：
- 這次只先收材料
- 還沒決定最後交給誰
- 想等人工再挑收斂工具

## 整合器選擇原則

### 需要長期延續的結果

優先 `cowork`。

### 需要快速得到一份比較報告

優先 `code`。

### 先收資料再決定

選 `manual`。

## 使用指令範例

```bash
_agent/skills/multi-ai-dispatcher/scripts/cross-check.sh --mode outside 教案.md
```

```bash
_agent/skills/multi-ai-dispatcher/scripts/cross-check.sh --mode style 文章.md --prompt "幫這篇文章想 5 個標題"
```

```bash
_agent/skills/multi-ai-dispatcher/scripts/cross-check.sh --mode blindspot 會議記錄.md --integrator code
```

## 設計原則

不要把最後整合綁死在 shell script。

原因很簡單：
- shell 適合穩定調度
- 真正的判斷與收斂，還是交給最適合當下任務的 AI

所以這份技能包的腳本定位，是「材料包產生器」，不是「唯一結論產生器」。
